# GLiNER2 Label Schema — Atman

**Статус:** финал T1 (HLE-378)
**Дата:** 2026-05-30
**Основано на:** read-only инвентаризации кода (T0 / HLE-409, `gliner2_label_inventory.md`)
**Базовая модель:** `fastino/gliner2-multi-v1` (mDeBERTa-v3-base, 100+ языков)

Документ фиксирует итоговый набор меток для дообучения GLiNER2 под русский, разбитый
по четырём адаптерам. Каждый адаптер — отдельный LoRA, ≤ 25 меток в NER/relation-режиме
и ≤ 50 в classification-режиме.

Принцип отбора: маленькие закрытые наборы, только то, что **реально извлекается из
текста**. Внутренние классификаторы и непрерывные метрики в GLiNER2 НЕ выносятся.
Решения о составе принимались под чистую базу (старые данные — мусор с нерабочих
прогонов, сохранять нечего), поэтому закрытые наборы можно сразу зашивать в SQL CHECK.

---

## Сводка адаптеров

| Адаптер | Режим | Меток | Бюджет |
|---|---|---|---|
| `atman-ner-core` | NER | 13 | ≤ 25 |
| `atman-relations` | relations | 12 | ≤ 25 |
| `atman-affect` | classification + span | 10 + 3 + 1 | ≤ 50 |
| `atman-pii-ru` | NER | 8 | ≤ 25 |

Все четыре с большим запасом в пределах лимитов GLiNER2.

## Адаптер 1 — `atman-ner-core` (сущности)

NER-режим. 13 меток. Таксономия спроектирована с нуля — в коде Atman NER-типов нет
(`FactRecord.tags` был free-form `list[str]`, никакого `EntityType` enum).

| Метка | Описание (entity description для обучения) | Примеры |
|---|---|---|
| `person` | Разумный субъект: человек, сам агент Atman, другие агенты | Маша, брат, начальник, Atman, агент |
| `organization` | Компания, учреждение, заведение | Яндекс, университет, поликлиника |
| `location` | Физическое место | Москва, дома, на работе |
| `date_time` | Время: дата, период, момент | вчера, в 2021, по утрам |
| `event` | Событие в жизни | свадьба, увольнение, переезд |
| `project` | Проект, начинание | Atman, ремонт, диссертация |
| `product` | Вещь, продукт, инструмент | телефон, Postgres, книга |
| `activity` | Занятие, хобби, действие | бег, программирование, готовка |
| `profession` | Профессия, роль | разработчик, врач, студент |
| `health` | Здоровье: состояние, симптом, лекарство | простуда, головная боль, аспирин |
| `emotion_word` | Прямо названная эмоция | злюсь, рад, тревожно |
| `money` | Деньги, суммы | зарплата, 1000 рублей, долг |
| `animal` | Питомцы, животные | кот, собака Рекс |

**Заметка по `person`:** метка покрывает и людей, и агентов (Atman, другие агенты) —
из текста они ведут себя в графе одинаково. Различение «я / другой агент / человек»
выносится в резолвер сущностей Atman (HLE-642), НЕ в GLiNER2. Местоимения («я», «он»)
GLiNER2 не разрешает — это работа резолвера по автору сообщения.

**Триггер на расширение:** если на тестах появятся факты / key moments без привязки
к сущностям — значит категорий не хватает, добавляем по факту того, что реально
теряется. Заранее не гадаем.

## Адаптер 2 — `atman-relations` (отношения)

Relation-режим. 12 отношений. Спроектировано с нуля — заменяет выкинутый mREBEL.
Системные `superseded_by`/`supersedes` НЕ входят (генерируются движком при суперседе
фактов, не извлекаются из текста).

| Отношение | Направление | Пример |
|---|---|---|
| `works_at` | субъект → организация | Маша работает в Яндексе |
| `lives_in` | субъект → место | живу в Москве |
| `located_in` | объект → место | офис в центре |
| `knows` | субъект → субъект (знакомство) | знаком с Алексеем |
| `related_to` | субъект → субъект (родство/близость) | брат, жена, родня |
| `part_of` | объект → большее целое | отдел в компании |
| `participates_in` | субъект → событие/проект | участвую в Atman |
| `owns` | субъект → вещь | моя машина |
| `created` | субъект → проект/продукт | я сделал Atman |
| `happened_at` | событие → дата/место | свадьба была в 2020 |
| `causes` | объект → объект | стресс привёл к болезни |
| `feels_about` | субъект → объект | люблю свою работу |

**Заметки:**
- `knows` vs `related_to` — намеренно разделены (знакомый ≠ родственник). Различение
  бывает сложным из текста; проверяем на тестах, при больших путаницах — пересмотр.
- `feels_about` — на грани с affect-частью; оставлено намеренно, ловит отношение к
  объектам. Эмоции как таковые — в `atman-affect`.
- «субъект» = `person` в смысле адаптера 1 (человек / Atman / агент).

## Адаптер 3 — `atman-affect` (мотивации, глубина, отказ)

Тонкий адаптер. Только то, что извлекается из текста. Composite-схема:
2 classification-задачи + 1 span.

### 3a. `motivation` (classification, 10) — система Шварца

«Что движет субъектом» в моменте. Объективное наблюдение из текста. Мульти-лейбл,
порог 0.3–0.4 (полнота важнее точности).

| Мотивация | Смысл |
|---|---|
| `self_direction` | независимость, свобода выбора |
| `stimulation` | новизна, азарт, движение |
| `hedonism` | удовольствие, наслаждение |
| `achievement` | успех, компетентность, признание |
| `power` | влияние, статус, контроль |
| `security` | стабильность, безопасность, порядок |
| `conformity` | следование правилам, не подводить |
| `tradition` | уважение к традициям, обычаям |
| `benevolence` | забота о близких, доброта |
| `universalism` | справедливость, забота о мире и людях |

**КРИТИЧНО — терминология.** Это `motivation` («что движет»), а НЕ «личные ценности
агента». Личные ценности/принципы агента — freeform, агент формулирует их сам и сам
приоритизирует; они живут в рефлексии, GLiNER2 их НЕ извлекает. Разделение двух понятий
и схема БД под него — тикеты HLE-643 (логика рефлексии) и HLE-644 (схема БД: добавить
`motivation`, расщепить `values_touched`). Переименование Шварц→`motivation` сделано
именно чтобы развести с личными ценностями.

### 3b. `emotional_depth` (classification, 3)

Уже закрытый enum в коде (`core/models/experience.py:34-36`, SQL CHECK в
`0004_agent_schema.sql`). Single-label.

| Метка | Смысл |
|---|---|
| `surface` | поверхностно |
| `meaningful` | значимо |
| `profound` | глубоко |

### 3c. `value_refusal` (span / binary)

Отказ по ценностным/моральным соображениям (не «не могу», а «не буду — против
принципов»). В коде есть детектор RU/EN (`affect/refusal_detector.py`,
disgust/anger-сигнал как дискриминатор vs обычный отказ по неспособности).

**Что НЕ входит в адаптер (остаётся на LLM / существующих классификаторах):**
`moral_orientation`, `helpfulness_level`, `pattern_type`, `jahoda_criterion` —
внутренние классификаторы рефлексии, не извлекаются из сырого текста.
`NRC emotion` (10 каналов), `affect_metrics` (8 полей) — непрерывные числа, не метки.

## Адаптер 4 — `atman-pii-ru` (чувствительные данные)

NER-режим. 8 меток. С нуля — PII в Atman отсутствует полностью (grep подтвердил ноль).
Fastino-овский PII-фильтр русский не поддерживает. Цель — замечать чувствительное,
чтобы Atman не светил это в логах и не писал в память.

| Метка | Описание | Regex-валидатор |
|---|---|---|
| `phone` | Телефонный номер | `+7 (XXX) XXX-XX-XX` и варианты |
| `email` | Адрес почты | стандартный email |
| `address` | Почтовый адрес | — (свободная форма) |
| `date_of_birth` | Дата рождения | дата |
| `passport_rf` | Паспорт РФ | `XXXX XXXXXX` (серия + номер) |
| `snils` | СНИЛС | `XXX-XXX-XXX XX` |
| `inn` | ИНН | 10 или 12 цифр |
| `credential` | Секреты: пароли, API-ключи, токены, SSH-ключи | префиксы `sk-`, `ghp_`, `AKIA`, `-----BEGIN ... KEY-----`, base64 |

**Решения T1:**
- `full_name` НЕ берём — дублирует `person` из NER-адаптера.
- `bank_card` НЕ берём — на всякий случай не пишем в память вообще.
- `ip_address` выкинут — не секрет, виден в сети.
- `ogrn` выкинут — это про юрлицо, не личные данные человека.
- `credential` — одна метка на все типы секретов (не дробим на password / api_key /
  ssh_key / token); при необходимости тип уточняется regex'ом. Recall критичен (≥ 0.95) —
  пропущенный секрет утечёт в память/логи.

**Важно не путать:** детекция `credential` (формат секрета в тексте) ≠ фича «агент не
разглашает чужое из личных бесед» (HLE-645). Первое — про формат данных, второе — про
границы раскрытия между собеседниками.

## Сопутствующие решения (вынесены из GLiNER2-проекта)

Дизайн схемы вскрыл вопросы за пределами fine-tune. Заведены отдельными тикетами:

| Тикет | Суть | Где |
|---|---|---|
| HLE-633 | Баг: `reflection_type="general"` отлетает по SQL CHECK | Atman баги |
| HLE-642 | Резолвер субъекта: «я»/местоимения → сущность по автору сообщения | Atman баги |
| HLE-643 | Рефлексия: развести «мотивацию» (Шварц) и личные ценности агента | Atman баги |
| HLE-644 | Схема БД: добавить `motivation`, расщепить `values_touched` | Atman баги |
| HLE-645 | Приватность: агент не разглашает чужое из личных бесед | Atman фичи |

## Регистр и формат меток

- Метки латиницей, lower_snake_case (как в таблицах выше) — единообразно для обучающих
  данных и инференса.
- Каждой метке при обучении даётся entity description (колонка «Описание») — GLiNER2
  использует их как подсказки, повышает качество на русском.
- Обучающие данные и вызовы инференса используют ОДИН регистр и ОДНИ строки меток.
