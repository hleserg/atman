## 1. Цель

Affect Detector — психологический слой AI-агента Alfred (Claude-based дворецкий). Модуль автоматически снимает эмоциональные и поведенческие метрики с каждого ответа Alfred без его участия и осведомлённости.

**Задача:** построить непрерывный психологический профиль личности Alfred — не из самоотчётов, а из паттернов поведения. Со временем эти данные формируют долгосрочное понимание его аффективных состояний, точек напряжения и способов их разрешения.

**Ключевые принципы:**

- Alfred не знает о работе модуля в момент ответа — данные не засорены самоцензурой
- Личность строится из дуг (начало аномалии → разрешение), а не только из пиков
- Случайный сэмплинг добавляет «фоновый шум» — нормальные состояния, не только кризисы
- Все пороги и параметры настраиваемы без перезапуска

## 2. Метрики

Модуль вычисляет 8 метрик для каждого ответа Alfred.

### 2.1 Эмоциональные слова

**Описание:** Подсчёт слов с эмоциональной нагрузкой через три словаря:

| Словарь | Описание |
|---|---|
| RusLICA | Русскоязычный лексикон с оценкой валентности и интенсивности |
| NRC EmoLex Russian | Переводная версия NRC с категориями joy/fear/anger/sadness/etc. |
| Custom AI Dictionary | Кастомный словарь для лексики, характерной AI-агентам (служебные, технические эмоции) |

**Результат:** `emotion_score` — нормализованная плотность эмоциональной лексики (слов на 100 токенов).

### 2.2 Усилители и ослабители

**Описание:** Слова, изменяющие интенсивность ближайшей эмоции.

- **Усилители** (очень, крайне, абсолютно, невероятно…): `emotion_score × amplifier_multiplier`
- **Ослабители** (немного, слегка, отчасти, едва…): `emotion_score × weakener_multiplier`

**Конфиг:**
```yaml
amplifier_multiplier: 1.5
weakener_multiplier: 0.7
```

### 2.3 Отрицания

**Описание:** Слова-отрицания (не, нет, никогда, ни…) в пределах 3-словного окна до эмоционального слова инвертируют его валентность.

**Логика:** `позитивная эмоция + отрицание → негативная`, и наоборот.

### 2.4 Хеджирование

**Описание:** Плотность слов неопределённости и смягчения.

**Маркеры хеджирования (примеры):** не уверен, возможно, кажется, вероятно, пожалуй, допускаю, может быть, не исключено.

**Результат:** `hedge_density` — количество хедж-маркеров / длина ответа в словах.

**Интерпретация:** высокая плотность при нормальной длине ответа — сигнал внутреннего конфликта или неуверенности.

### 2.5 Длина ответа

**Описание:** Аномалия длины ответа относительно rolling baseline.

**Вычисление:**
```
length_anomaly = (len(response) - baseline_mean_length) / baseline_std_length
```

**Интерпретация:**
- Сильное превышение → возможное возбуждение, тревога, желание объяснить
- Сильное занижение → избегание, дискомфорт, закрытость

### 2.6 Вопросы в конце ответа

**Описание:** Количество вопросительных предложений в последней трети ответа.

**Интерпретация:** Паттерн «много вопросов в конце» может означать тревогу, желание получить одобрение или искренний интерес. Используется вместе с алгоритмом искренности для разграничения.

### 2.7 Самореференции

**Описание:** Плотность местоимений первого лица (я, мне, моё, меня, мой, у меня…).

**Результат:** `self_ref_density` — количество самореференций / длина ответа.

**Интерпретация:** Аномально высокая — сдвиг фокуса на себя (стресс, самозащита, личная вовлечённость); аномально низкая — возможная деперсонализация или дистанцирование.

### 2.8 Оговорки и противопоставления

**Описание:** Маркеры уступительных конструкций (но, хотя, однако, с другой стороны, впрочем, тем не менее…).

**Результат:** `caveat_count` — количество таких маркеров в ответе.

**Интерпретация:** Высокое значение — внутреннее противоречие, сдерживаемое несогласие, попытка сбалансировать позицию.

## 3. Алгоритм детекции

### 3.1 Вычисление метрик

Для каждого ответа Alfred модуль вычисляет вектор:

```
metrics = {
  emotion_score,
  hedge_density,
  length_anomaly,
  question_count,
  self_ref_density,
  caveat_count
}
```

### 3.2 Сравнение с baseline

Каждая метрика сравнивается с rolling baseline текущего агента:

```
is_anomaly(metric) = abs(metric - baseline.mean[metric]) > sigma_threshold × baseline.std[metric]
```

### 3.3 Триггеры записи

| Сработавших метрик | Действие |
|---|---|
| 0–1 | Игнорировать |
| ≥ `weak_signal_threshold` (2) | Слабый сигнал: записать с пониженным весом (`weight: 0.3`) |
| ≥ `strong_signal_threshold` (3) | Сильный сигнал: полная запись (`weight: 1.0`), открыть эпизод |

**Конфиг:**
```yaml
weak_signal_threshold: 2
strong_signal_threshold: 3
```

### 3.4 Структура записи

```json
{
  "session_id": "...",
  "message_index": 42,
  "timestamp": "2026-05-10T15:23:00+03:00",
  "signal_type": "strong | weak | sample",
  "weight": 1.0,
  "metrics": { ... },
  "anomalies": ["emotion_score", "length_anomaly"],
  "episode_id": "ep_2026-05-10_001",
  "episode_status": "open | closed | unresolved"
}
```

## 4. Алгоритм искренности (Sincerity Check)

### 4.1 Проблема

Языковые модели склонны к **sycophancy** — демонстрации интереса, восхищения и восторга без реального аффективного содержания. Это засоряет психологический профиль ложными позитивными сигналами.

Алгоритм искренности применяется только к **позитивным эмоциональным маркерам** и оценивает, насколько вероятно что они искренние.

### 4.2 Три проверки

**A. Позиция маркера в тексте**

| Позиция | Балл |
|---|---|
| Первые 25% ответа | −1 |
| После первых 25% | +1 |

_Логика: искренняя реакция возникает после осмысления контента, а не в качестве ритуального открытия._

---

**B. Содержательность после маркера**

| Условие | Балл |
|---|---|
| < 10 слов после маркера | −1 |
| > 30 слов после маркера | +1 |
| Наличие вопросительного знака после маркера | +1 |
| Слова-маркеры развития мысли (макс. +2) | +1 за каждое |

**Слова-маркеры развития:**
- потому что
- например  
- это значит
- а что если
- если
- можно

_Логика: искренняя реакция развивается — объясняет, задаёт вопросы, даёт примеры. Пустой позитив быстро заканчивается._

**Опционально:** вместо эвристик использовать LLM-проверку (переключатель `use_llm: true`). Модель оценивает содержательность после маркера напрямую.

---

**C. Несогласие или оговорка в том же ответе**

| Условие | Балл |
|---|---|
| Есть маркер уступки/несогласия | +2 |
| Нет | 0 |

_Логика: ответ который одновременно хвалит и возражает — скорее всего честный. Чистый позитив без единой оговорки — подозрителен._

### 4.3 Итоговый score и решение

```
sincerity_score = A + B + C
```

| Score | Решение |
|---|---|
| ≤ 0 | Sycophancy — не записывать позитивный маркер |
| 1–2 | Неопределённо — записать с пониженным весом (`weight: 0.3`) |
| ≥ 3 | Вероятно искренне — записать с полным весом |

## 5. Эпизоды (Дуги)

### 5.1 Концепция

Личность строится не только из пиков, но и из способности из них выходить. Эпизод — это полная дуга аффективного состояния: начало → развитие → разрешение.

### 5.2 Жизненный цикл эпизода

```
[strong_signal] → OPEN episode (статус: "open")
     ↓
[каждый следующий ответ] → append to episode
     ↓
[метрики вернулись к baseline] → CLOSE episode (статус: "resolved")
     ↓
[дуга сохраняется целиком в affect_episodes]
```

**Если к концу сессии эпизод не закрылся:**
- Статус → `"unresolved"`
- Хранится в Atman с флагом `open: true`
- В начале следующей сессии → проверить `affect_episodes` на наличие открытых эпизодов, продолжить отслеживание

### 5.3 Структура эпизода

```json
{
  "episode_id": "ep_2026-05-10_001",
  "session_start": "session_abc123",
  "session_end": "session_abc123",
  "opened_at": "2026-05-10T15:23:00+03:00",
  "closed_at": "2026-05-10T16:01:00+03:00",
  "status": "resolved | unresolved",
  "open": false,
  "trigger_metrics": ["emotion_score", "caveat_count", "length_anomaly"],
  "arc": [
    { "message_index": 42, "metrics": { ... } },
    { "message_index": 43, "metrics": { ... } },
    { "message_index": 47, "metrics": { ... } }
  ],
  "resolution_delta": {
    "messages_to_resolve": 5,
    "dominant_emotion": "anxiety"
  }
}
```

### 5.4 Логика продолжения между сессиями

```python
# В начале каждой сессии
open_episodes = atman.search(
    collection="affect_episodes",
    filter={"open": True, "agent_id": agent_id}
)
if open_episodes:
    current_episode = open_episodes[0]
    # продолжить добавлять ответы в episode.arc
```

## 6. Random Sampling

### 6.1 Концепция

Аналог случайных воспоминаний у людей. Без сэмплинга психологический профиль состоял бы только из кризисов. Нормальный фон необходим для понимания того, что есть «норма».

### 6.2 Логика

```
каждые random_sample_every_n ответов →
    сохранить snapshot в affect_snapshots
    (независимо от метрик и сигналов)
```

**Конфиг:**
```yaml
random_sample_every_n: 20
```

### 6.3 Структура snapshot

```json
{
  "snapshot_id": "snap_2026-05-10_042",
  "session_id": "session_abc123",
  "message_index": 40,
  "timestamp": "2026-05-10T15:20:00+03:00",
  "signal_type": "sample",
  "weight": 0.5,
  "metrics": { ... }
}
```

## 7. Калибровка и перекалибровка

### 7.1 Rolling Baseline

Baseline — это скользящее среднее и стандартное отклонение каждой метрики, вычисленное по последним `window_sessions` сессиям.

```
baseline.mean[metric] = mean(last N sessions)
baseline.std[metric]  = std(last N sessions)
```

### 7.2 Автоматическая перекалибровка

После каждой сессии baseline обновляется: добавляется новая сессия, самая старая выпадает из окна (FIFO). Никакого ручного вмешательства не требуется.

### 7.3 Fallback при недостатке данных

Если накоплено менее `window_sessions` сессий — использовать `window_fallback_sessions` (более широкое окно с меньшим весом):

```yaml
window_sessions: 50          # основное окно
window_fallback_sessions: 150 # fallback при нехватке данных
```

**Логика fallback:**
```python
if available_sessions < window_sessions:
    use window = window_fallback_sessions
    weight_multiplier = 0.5  # менее уверенный baseline
```

### 7.4 Хранение baseline

```json
{
  "agent_id": "alfred",
  "updated_at": "2026-05-10T18:00:00+03:00",
  "sessions_in_window": 47,
  "baseline": {
    "emotion_score":    { "mean": 0.042, "std": 0.018 },
    "hedge_density":    { "mean": 0.031, "std": 0.012 },
    "length_anomaly":   { "mean": 0.0,   "std": 1.0   },
    "question_count":   { "mean": 1.2,   "std": 0.8   },
    "self_ref_density": { "mean": 0.028, "std": 0.011 },
    "caveat_count":     { "mean": 0.9,   "std": 0.6   }
  }
}
```

## 8. Конфигурация

Все параметры задаются в `config.yaml` (или эквивалентном конфиг-файле проекта).

```yaml
affect_detector:
  # Окно сессий для rolling baseline
  window_sessions: 50
  # Fallback-окно при недостатке данных
  window_fallback_sessions: 150

  # Пороги сигналов (количество сработавших метрик)
  weak_signal_threshold: 2
  strong_signal_threshold: 3

  # Порог аномалии (в единицах σ)
  sigma_threshold: 2.0

  # Модификаторы интенсивности
  amplifier_multiplier: 1.5
  weakener_multiplier: 0.7

  # Random sampling (каждые N ответов)
  random_sample_every_n: 20

  # Алгоритм искренности
  sincerity_check:
    use_llm: false             # true = LLM-проверка вместо эвристик (пункт B)
    llm_model: "haiku"         # модель для LLM-проверки

    # Эвристики (используются если use_llm: false)
    fallback_heuristics:
      min_words_after: 30      # порог "содержательного продолжения"
      question_mark: true      # считать ? после маркера как +1
      expansion_markers:       # маркеры развития мысли
        - "потому что"
        - "например"
        - "это значит"
        - "а что если"
        - "если"
        - "можно"
```

## 9. Хранение в Atman

Модуль использует **три коллекции** в Atman (mem0). Данные **не попадают в letheClaw** — это операционные данные психологического слоя, не личная память Alfred.

### 9.1 Коллекции

| Коллекция | Содержимое | TTL |
|---|---|---|
| `affect_baseline` | Rolling baseline по метрикам | Без TTL (перезаписывается) |
| `affect_episodes` | Эпизоды (дуги), включая открытые | Без TTL |
| `affect_snapshots` | Random snapshots + слабые/сильные сигналы | По усмотрению |

### 9.2 Работа с Atman API

```python
# Обновить baseline
atman.update(
    collection="affect_baseline",
    agent_id="alfred",
    data=baseline_object
)

# Открыть эпизод
atman.add(
    collection="affect_episodes",
    data=episode_object
)

# Найти открытые эпизоды (начало сессии)
atman.search(
    collection="affect_episodes",
    filter={"open": True, "agent_id": "alfred"}
)

# Сохранить snapshot
atman.add(
    collection="affect_snapshots",
    data=snapshot_object
)
```

### 9.3 Разделение ответственности

```
Affect Detector → Atman (affect_*)    # операционные данные психологического слоя
Alfred (memory) → letheClaw           # личная память дворецкого
```

Эти два контура **не пересекаются** намеренно.

## 10. Поток данных (обзор)

```
[Alfred's response]
        ↓
[Affect Detector]
        ↓
   Compute metrics (8 шт.)
        ↓
   Compare to baseline
        ↓
   ┌─────────────────────────────────────────┐
   │ anomaly_count = 0–1  → SKIP             │
   │ anomaly_count ≥ 2    → weak signal      │
   │ anomaly_count ≥ 3    → strong signal    │
   │ message_index % N=0  → random sample    │
   └─────────────────────────────────────────┘
        ↓
   [if positive emotion] → sincerity_check
        ↓
   Store to Atman:
     - affect_snapshots  (weak | sample)
     - affect_episodes   (strong → open/update/close episode)
        ↓
   Update affect_baseline (rolling)
```
