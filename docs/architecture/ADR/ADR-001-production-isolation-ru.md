# ADR-001 — Изоляция продакшена (пространство имён eval)

> **English version:** [ADR-001-production-isolation.md](ADR-001-production-isolation.md)

**Статус:** Принято  
**Дата:** 2026-05-19

## Контекст

У Atman два типа пользователей с разными потребностями:

1. **Продакшен-пользователи** — запускают агентов в живых окружениях. Они устанавливают Atman для памяти и рефлексии агента. Им не нужна eval-инфраструктура: ни Alembic, ни SQLAlchemy, ни eval-специфичные схемы базы данных.

2. **Исследователи и разработчики** — запускают eval-наборы, воспроизводят сессии, сравнивают конфигурации. Им нужна тяжёлая eval-инфраструктура (миграции баз данных, большие зависимости моделей).

Смешивание этих двух профилей в одной установке означает, что каждое продакшен-развёртывание несёт eval-зависимости. Это раздувает размер контейнера, увеличивает поверхность атаки и создаёт ненужную связность.

## Решение

Atman использует **один репозиторий с двумя профилями установки**, реализованными на уровне Python-пакетирования:

| Профиль | Команда | Содержимое |
|---------|---------|-----------|
| Продакшен | `pip install atman` | Только `core/` + `adapters/` |
| Eval | `pip install "atman[eval]"` | Core + Alembic, SQLAlchemy, PostgreSQL |
| Dev | `pip install "atman[dev]"` | Core + инструменты тестирования и линтинга |
| Всё | `pip install "atman[all]"` | Всё вместе |

**Граница модулей:**

- `src/atman/core/` и `src/atman/adapters/` — продакшен-код. Без импорта eval-модулей.
- `src/atman/eval/` — только eval-код. Изолирован за lazy import guard в `eval/__init__.py`. Продакшен-код никогда не импортирует его.

**Граница базы данных:**

- Продакшен-схема: таблицы `public.*`, управляемые продакшен-миграциями.
- Eval-схема: таблицы `public.eval.*`, управляемые `eval/migrations/` (Alembic), загружаются только с `atman[eval]`.

**Контракт линтера импортов:**

В CI применяется правило `no-core-to-optional-agent-cli`. Оно падает, если любой модуль из `core/` или `adapters/` импортирует из `eval/`.

## Последствия

**Положительные:**
- Продакшен-установки лёгкие. Никаких eval-зависимостей в продакшен-контейнерах.
- Eval-инфраструктура может развиваться независимо без риска для продакшен-рантайма.
- Чёткая граница импортов делает архитектурные нарушения обнаруживаемыми во время CI.

**Отрицательные:**
- Два трека миграций для поддержки (продакшен + eval).
- Разработчики должны осознавать, в каком профиле они работают.
- Lazy import guard добавляет небольшое количество бойлерплейта в `eval/__init__.py`.

## Влияние на миграцию

Существующий код, который импортировал eval-утилиты напрямую из не-eval-модулей, должен быть рефакторен для использования паттерна lazy import или перемещён в `eval/`. Любое такое нарушение обнаруживается линтером импортов перед мержем.
