# Инструкция для coding agents

Этот файл обязателен для всех агентов, которые пишут код в репозитории Atman.
Если агент получает отдельный work package, он всё равно должен сначала прочитать этот файл.

## 1. Что читать перед работой

Перед изменениями прочитай:

1. `AGENTS.md` - общие правила разработки.
2. `docs/architecture/SYSTEM.md` - архитектурный источник правды.
3. Свой work package из `docs/development/work-packages/`.
4. Существующий код и тесты в области, которую меняешь.

Если требования work package конфликтуют с `SYSTEM.md`, не маскируй конфликт в коде. Зафиксируй его в PR как открытый вопрос и выбери минимальное безопасное поведение.

## 2. Рабочая единица

Каждый work package должен давать вертикальный, запускаемый результат:

- доменные модели;
- минимальную реализацию;
- локальный adapter или storage;
- CLI/API/demo для ручного запуска;
- тесты;
- README или раздел документации с командами запуска.

Нельзя сдавать только схемы, только абстракции или только "подготовку к будущей интеграции".

## 3. Границы пакета

Делай только свой пакет и минимальные общие контракты, которые нужны для его запуска.

Разрешено:

- создать локальный порт, если соседний пакет ещё не реализован;
- использовать in-memory/file adapters для тестов;
- добавить fixtures;
- добавить маленький общий helper, если без него пакет не запускается.

Не разрешено:

- реализовывать соседний work package "по пути";
- завязывать unit-тесты на mem0, OpenClaw, внешний LLM, scheduler или API key;
- добавлять глобальную архитектуру, которая не требуется текущему runnable-срезу;
- ретроактивно менять смысл уже сохранённого опыта.

## 4. Архитектурные инварианты Atman

Эти правила важнее локального удобства реализации.

### Factual Memory

- Хранит проверяемые факты и связи.
- Не хранит эмоции, принципы, привычки или интерпретации как факты.
- Не смешивает наблюдение и вывод.

### Experience Store

- Хранит first-person experience, а не сырой лог и не анализ.
- `how_i_felt` записывается в момент события или помечается `incomplete_coloring`.
- Оригинальные `key_moments` append-only/immutable.
- `reframing_notes` добавляются слоями и не переписывают исходное переживание.
- `salience` может угасать, но запись опыта не должна терять исходный смысл.

### Reflection Engine

- Работает только с уже окрашенным опытом.
- Не придумывает эмоции задним числом.
- Различает факты, интерпретации, привычки, принципы и навыки.
- Результаты рефлексии записывает как отдельные события/заметки.

### Identity и Narrative

- Bootstrap identity не должна содержать выдуманные seed-принципы.
- Identity версионируется через snapshots.
- `NARRATIVE.md` пишется от первого лица и читается первым при загрузке личности.
- `CORE LAYER` обновляется редко и только при явном основании.
- `THREADS` не удаляются молча: закрытие должно быть явным.

### Session Manager

- Experience Store получает уже окрашенный опыт, не сырой лог.
- Неполная окраска должна быть видна явно.
- Сессионный runtime не должен подменять Reflection Engine.

### Reality Anchor и Affect

- Любой drift/intervention должен иметь объяснимое evidence.
- Пороги должны быть настраиваемыми.
- Добросовестная ошибка и галлюцинация не считаются одним и тем же без явного основания.

### Skills

- Навык не является принципом.
- Disabled skill нельзя восстанавливать самовольно.
- Passive/core skills не удаляются garbage collection.

## 5. Технические соглашения

- Предпочитай простые порты: `storage`, `memory`, `model`, `workspace`, `clock`.
- Для тестов используй deterministic providers и fixtures.
- Любой внешний сервис должен иметь мок или file/in-memory замену.
- CLI/demo должен работать из чистого checkout по описанной команде.
- Ошибки должны быть диагностируемыми: возвращай понятные сообщения и не теряй входные данные.
- Сохраняй данные в читаемых форматах, если это не противоречит задаче: JSON, JSONL, TOML, Markdown.
- Не добавляй тяжёлые зависимости без реальной необходимости.
- Не вводи фоновые бесконечные процессы как обязательное условие теста.

## 6. Тестирование

Минимум для каждого пакета:

- unit-тесты доменных инвариантов;
- тесты adapters на in-memory/file storage;
- smoke test CLI/demo;
- негативные тесты на запрещённое поведение из `SYSTEM.md`.

Если пакет пока не может интегрироваться с соседними компонентами, используй fixtures с теми же полями, которые описаны в work package.

## 7. Документация результата

В PR обязательно укажи:

- какой work package реализован;
- какие команды запускают demo/CLI;
- какие команды запускают тесты;
- какие внешние сервисы не нужны для локальной проверки;
- какие ограничения или открытые вопросы остались.

Если добавлен новый контракт, коротко опиши:

- кто его вызывает;
- какие поля обязательны;
- какие инварианты он сохраняет;
- как его заменить реальным adapter в будущем.

## 8. Definition of Done

Работа готова только если:

- результат запускается локально;
- тесты проходят;
- work package не расширен за пределы своей ответственности;
- архитектурные инварианты не нарушены;
- README/PR содержит команды проверки;
- временные fixtures, demo-data и generated files не засоряют рабочую директорию.

## 9. Как формулировать задачу агенту

Минимальный обязательный prompt для coding agent:

```text
Прочитай AGENTS.md, docs/architecture/SYSTEM.md и work package <путь>.
Реализуй только этот work package как самостоятельный runnable-срез.
Не жди готовности соседних пакетов: используй локальные порты, fixtures и in-memory/file adapters.
Не подключай внешние сервисы в тестах.
В конце добавь команды запуска и тестов в README/PR.
```
